# Lemma: Zero audit records on failure or rejection

-----------------------------------------
**Lemma ID:** LEM-M365-E-002-01

**Invariant:** INV-M365-E-002

**Derived from:**
- Section 5: Audit Event Semantics (no audit on failure or rejection)
- Section 6: Determinism and Failure Constraints

**Statement:**

Whenever \(\mathrm{Eval}(x, \tau)\) is undefined (rejected/invalid), no audit record is produced: there is no output triple, so no \(\mathit{audit}\) is emitted; the requirement is that failed or rejected instructions produce zero audit records. Whenever \(\mathrm{Eval}(x, \tau)\) is defined but \(r\) indicates failure, \(\mathit{audit} = ()\) (empty sequence), so \(|\mathit{audit}| = 0\).

**Assumptions:**

- Section 5: "If \(\mathrm{Eval}(x, \tau)\) is undefined (rejected/invalid), or if it is defined with \(r\) indicating failure: \(\mathit{audit} = ()\). So failed or rejected instructions produce zero audit records."
- Section 6: when \(\mathrm{Eval}(x, \tau)\) is undefined, no state change and no audit are produced.

**Proof outline:**

1. **Case 1: Rejection (undefined).** Suppose \(\mathrm{Eval}(x, \tau)\) is undefined. Then there is no \((r, \tau', \mathit{audit})\) such that \(\mathrm{Eval}(x, \tau) = (r, \tau', \mathit{audit})\). The system does not produce an output; Section 6 states that in this case no audit records are produced. So the number of audit records attributable to this evaluation is zero.

2. **Case 2: Defined but failure.** Suppose \(\mathrm{Eval}(x, \tau)\) is defined and the result \(r\) indicates failure. Then \(\mathrm{Eval}(x, \tau) = (r, \tau', \mathit{audit})\) for some \((r, \tau', \mathit{audit})\). Section 5 explicitly states that when the result indicates failure, \(\mathit{audit} = ()\). The empty sequence has length zero, so \(|\mathit{audit}| = 0\).

3. **Combined:** In both cases—rejection (undefined) and defined failure—the specification requires zero audit records. Hence whenever evaluation is rejected or the result indicates failure, \(|\mathit{audit}| = 0\).

**Edge cases:**

- "Rejected" means \(\mathrm{Eval}(x, \tau)\) is undefined (e.g. \(x \notin \mathcal{X}_{\mathrm{valid}}\) or \(\mathrm{Valid}(x, \tau)\) false per Section 6). No triple is returned; the requirement that no audit is produced is stated in Section 6.
- The boundary between success and failure is given by "\(r\) indicates success" vs "\(r\) indicates failure"; Phase 1 leaves the exact encoding to the invariant phase but requires that in the failure case \(\mathit{audit} = ()\).

**Conclusion:**

Section 5 and Section 6 together require that failed or rejected instructions produce no audit records (empty sequence when defined with failure, no output when undefined). Hence INV-M365-E-002 holds.
