# Lemma: Persona, identity, action, and parameters are from the instruction tuple only

-----------------------------------------
**Lemma ID:** LEM-M365-G-001-01

**Invariant:** INV-M365-G-001

**Derived from:**
- Section 7: Explicit Non-Goals (no intent or routing; no identity resolution; instruction completeness)

**Statement:**

The values used for evaluation—persona \(p\), identity \(i\), action \(a\), and parameters \(\mathit{params}\)—are exactly those supplied in the instruction tuple \(x = (p, i, a, \mathit{params})\). M365 does not infer, choose, or override them. Formally: the only inputs to \(\mathrm{Eval}\) are \((x, \tau)\) with \(x \in \mathcal{X}\); the first component of \(x\) is \(p\), the second is \(i\), the third is \(a\), the fourth is \(\mathit{params}\); and the specification states that M365 does not decide intent, routing, or persona, and does not resolve identity.

**Assumptions:**

- Section 3: \(\mathrm{Eval}: \mathcal{X} \times \mathcal{T} \rightharpoonup \mathcal{R} \times \mathcal{T} \times \mathcal{E}^*\); the domain is \(\mathcal{X} \times \mathcal{T}\), so the only input describing the instruction is \(x \in \mathcal{X}\).
- Section 7: "M365 does not infer intent, choose persona, or route instructions. It only evaluates the given instruction tuple." "Identity \(i \in \mathcal{I}\) is taken as given. M365 does not authenticate, resolve, or validate identity; that is assumed done by the caller."
- Section 7: "Every executed instruction is fully described by \(x = (p, i, a, \mathit{params})\). There are no additional hidden inputs, side channels, or implicit context that affect \(\mathrm{Eval}\)."

**Proof outline:**

1. The domain of \(\mathrm{Eval}\) is \(\mathcal{X} \times \mathcal{T}\). So every evaluation is applied to a pair \((x, \tau)\) where \(x = (p, i, a, \mathit{params})\) is the instruction. The values \(p, i, a, \mathit{params}\) are therefore uniquely determined by \(x\) (they are the coordinates of the tuple).
2. Section 7 states that M365 does not infer intent, choose persona, or route instructions; it only evaluates the given instruction tuple. So the persona, action, and (by implication) the identity and parameters used for evaluation are not chosen or overridden by M365—they are the given tuple.
3. Section 7 states that identity is taken as given and M365 does not resolve or validate it. So the identity used is exactly the one in the instruction.
4. There is no other input in the domain of \(\mathrm{Eval}\) that could supply or override \(p, i, a, \mathit{params}\); the only instruction descriptor is \(x\). So the evaluation function uses instruction.persona, instruction.identity, instruction.action, and instruction.parameters as the sole source for those values.

**Edge cases:**

- If the specification allowed a second channel (e.g. "context" or "caller") to override the tuple, that would contradict Section 7 (no hidden context, instruction fully described by the tuple). Phase 1 excludes that.
- "Validated by CAIO" is an assumption about who calls M365; it does not mean M365 derives the tuple from elsewhere—the tuple is still the only input from the specification's perspective.

**Conclusion:**

The domain of \(\mathrm{Eval}\) and the non-goals in Section 7 imply that persona, identity, action, and parameters are taken exactly from the instruction tuple and are not inferred, chosen, or overridden by M365. Hence INV-M365-G-001 holds.
