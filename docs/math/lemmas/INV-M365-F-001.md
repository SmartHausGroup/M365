# Lemma: No state change on invalid or rejected evaluation

-----------------------------------------
**Lemma ID:** LEM-M365-F-001-01

**Invariant:** INV-M365-F-001

**Derived from:**
- Section 6: Determinism and Failure Constraints (failure closed / no side effects on invalid)

**Statement:**

Whenever \(\mathrm{Eval}(x, \tau)\) is undefined (instruction invalid or rejected), the tenant state remains \(\tau\) from the perspective of this system: no state change is produced. So the observable tenant state after a rejected evaluation equals the tenant state before the evaluation.

**Assumptions:**

- Section 6: "If \(x \notin \mathcal{X}_{\mathrm{valid}}\), or if \(\mathrm{Valid}(x, \tau)\) does not hold (when that predicate is defined), then \(\mathrm{Eval}(x, \tau)\) is **undefined**. In that case we require: No state change: tenant state remains \(\tau\) (from the perspective of this system). No audit: no audit records are produced. So invalid instructions 'fail closed' with no side effects."

**Proof outline:**

1. Suppose \(\mathrm{Eval}(x, \tau)\) is undefined. Then there is no \((r, \tau', \mathit{audit})\) such that \(\mathrm{Eval}(x, \tau) = (r, \tau', \mathit{audit})\).
2. The specification does not define any other mechanism by which the system updates tenant state. The only way the spec describes state change is via the codomain of \(\mathrm{Eval}\): when \(\mathrm{Eval}(x, \tau)\) is defined, the result includes \(\tau'\) (new tenant state). When \(\mathrm{Eval}(x, \tau)\) is undefined, there is no result, hence no \(\tau'\) produced by this system.
3. Section 6 explicitly requires: in the undefined case, "tenant state remains \(\tau\) (from the perspective of this system)". So the system does not mutate state when evaluation is undefined.
4. Therefore the observable tenant state after a rejected evaluation is the same as \(\tau\)—no state change attributable to this evaluation.

**Edge cases:**

- "Invalid" includes \(x \notin \mathcal{X}_{\mathrm{valid}}\) and, when defined, \(\mathrm{Valid}(x, \tau)\) false (Section 6). In all such cases \(\mathrm{Eval}(x, \tau)\) is undefined and the same conclusion holds.
- The spec speaks from "the perspective of this system"; external state changes (e.g. by other actors) are outside the scope of \(\mathrm{Eval}\).

**Conclusion:**

Section 6 requires that when \(\mathrm{Eval}(x, \tau)\) is undefined, no state change and no audit occur. Hence the tenant state remains \(\tau\) after a rejected or invalid evaluation. Therefore INV-M365-F-001 holds.
