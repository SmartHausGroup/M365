# Lemma: No hidden context; outputs depend only on (instruction, tenant_state)

-----------------------------------------
**Lemma ID:** LEM-M365-G-002-01

**Invariant:** INV-M365-G-002

**Derived from:**
- Section 3: Core Evaluation Function(s)
- Section 7: Explicit Non-Goals (no hidden context)

**Statement:**

The outcome of evaluation—i.e. the triple \((r, \tau', \mathit{audit})\) when \(\mathrm{Eval}(x, \tau)\) is defined—is determined solely by \((x, \tau)\). No other input (time, request ID, caller identity, or any hidden context) may affect the result, new tenant state, or audit events. Formally: \(\mathrm{Eval}\) is a partial function from \(\mathcal{X} \times \mathcal{T}\) to \(\mathcal{R} \times \mathcal{T} \times \mathcal{E}^*\); by definition, the value at \((x, \tau)\) depends only on that pair.

**Assumptions:**

- Section 3: \(\mathrm{Eval}: \mathcal{X} \times \mathcal{T} \rightharpoonup \mathcal{R} \times \mathcal{T} \times \mathcal{E}^*\) is the core map; its domain is \(\mathcal{X} \times \mathcal{T}\) and its codomain is \(\mathcal{R} \times \mathcal{T} \times \mathcal{E}^*\).
- Section 7: "Every executed instruction is fully described by \(x = (p, i, a, \mathit{params})\). There are no additional hidden inputs, side channels, or implicit context that affect \(\mathrm{Eval}\). So: instruction completeness." "No protocol or API detail" and "No operator or algorithm specification" reinforce that the abstract map is \((x, \tau) \mapsto (r, \tau', \mathit{audit})\) with no other inputs.

**Proof outline:**

1. **Function signature:** \(\mathrm{Eval}\) is defined as a partial function with domain \(\mathcal{X} \times \mathcal{T}\). In the mathematical sense, a function assigns to each point in its domain at most one value in its codomain; that value is determined only by the argument. So \((r, \tau', \mathit{audit})\)—when it exists—is determined only by \((x, \tau)\).

2. **No other inputs in the spec:** The specification does not introduce any other arguments to \(\mathrm{Eval}\). Section 7 explicitly states there are no additional hidden inputs, side channels, or implicit context that affect \(\mathrm{Eval}\). So any factor that would change the outcome without changing \((x, \tau)\) is excluded by the non-goals.

3. **Completeness:** Section 7 states that every executed instruction is fully described by the tuple and that the outcome is the abstract map \((x, \tau) \mapsto (r, \tau', \mathit{audit})\). So result, new_tenant_state, and audit_events are computable from \((instruction, tenant\_state)\) only; no additional inputs are used.

4. **Contrapositive:** If any output depended on an input other than instruction or tenant_state, then \(\mathrm{Eval}\) would not be a function of \(\mathcal{X} \times \mathcal{T}\) alone, contradicting the signature and Section 7.

**Edge cases:**

- Non-determinism would allow the same \((x, \tau)\) to yield different outputs, but that is already excluded by the partial function definition and Section 6 (determinism).
- "Time" or "request ID" affecting the result would constitute hidden context; Section 7 forbids that.

**Conclusion:**

The definition of \(\mathrm{Eval}\) as a partial function on \(\mathcal{X} \times \mathcal{T}\) and the explicit non-goal of no hidden context imply that the outcome of evaluation is determined solely by \((instruction, tenant\_state)\). Hence INV-M365-G-002 holds.
